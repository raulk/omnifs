# CLAUDE.md

## Project

omnifs is a projected filesystem that mirrors external services into local paths via FUSE. Providers are WASM components (wasm32-wasip1) that the host runtime loads and drives through a WIT interface.

Workspace members: `crates/cli`, `crates/host`, `providers/github`, `providers/dns`, `providers/test`.

## Build and test

```bash
just check          # fmt, clippy, test (host + providers)
just check-providers # clippy + test-compile for wasm32-wasip1 providers
just dev            # docker compose up --build -d (creates .secrets/github_token)
just shell          # docker exec into the running container
```

Host crates (cli, host) build for the native target. Providers build for `wasm32-wasip1`.

Provider clippy/test commands must include `--target wasm32-wasip1`:
```bash
cargo clippy -p omnifs-provider-dns --target wasm32-wasip1 -- -D warnings
cargo test -p omnifs-provider-dns --target wasm32-wasip1 --no-run
```

WASM tests compile but cannot execute on the host (no WASM runtime in the test harness). Use `--no-run` to verify compilation.

## Provider build pipeline

Providers are built in two steps, not with `cargo component build`. This is intentional.

**Step 1:** `cargo build --target wasm32-wasip1` compiles providers into core WASM modules. WIT bindings are generated by the `wit_bindgen::generate!` macro in each provider's `lib.rs`.

**Step 2:** `wasm-tools component new --adapt wasi_snapshot_preview1=build/wasi_snapshot_preview1.reactor.wasm` wraps each core module into a WASM component with a wasip1-to-wasip2 adapter.

The `build/wasi_snapshot_preview1.reactor.wasm` adapter must match the host's wasmtime version (currently v43). It was extracted from `wasi-preview1-component-adapter-provider` 43.0.1.

**Why not `cargo component build`?** It bundles an adapter from `wasi-preview1-component-adapter-provider` v29, which is ABI-incompatible with wasmtime 43. The v29 adapter's `cabi_import_realloc` hits an assertion failure when the host returns data to the guest (e.g., `random_get` for HashMap seed init). Using `wasm-tools` directly lets us supply the correct adapter.

**When upgrading wasmtime** to version N, regenerate the adapter:
```bash
# Create a throwaway crate, add wasi-preview1-component-adapter-provider = "N",
# and write WASI_SNAPSHOT_PREVIEW1_REACTOR_ADAPTER to build/wasi_snapshot_preview1.reactor.wasm
```

Do not reintroduce `cargo component build` unless its bundled adapter matches the host wasmtime version.

## Provider architecture

Each provider is a WASM component implementing the `omnifs:provider` WIT interface. The host calls three browse operations:
- `lookup_child(id, parent_path, name)` resolves a single path component (FUSE lookup)
- `list_children(id, path)` lists directory contents (FUSE readdir)
- `read_file(id, path)` reads file content (FUSE read)

Providers return either a terminal `ActionResult` or an `Effect` (HTTP fetch, git operation). The host executes the effect and calls `resume(id, result)` with the outcome. Providers store continuations keyed by correlation ID to resume from where they left off.

The host passes the `[config]` section of each provider's TOML file as serialized TOML bytes to `initialize()`. The `[config]` prefix is stripped; the provider receives bare keys (e.g., `default_resolver`, `[resolvers]`).

## Gotchas

**Projected file sizes must be non-zero.** The host translates `DirEntry.size: None` to 0 via `unwrap_or(0)`. The kernel sees a 0-byte file and never calls `read`. Use `size: Some(4096)` for projected files whose real size is unknown until read.

**`hashbrown::HashMap` vs `std::collections::HashMap` in providers.** Use `hashbrown` for provider-internal maps. It doesn't require `random_get` in WASI. Crate dependencies that internally use `std::collections::HashMap` (like `toml`, `regex`, etc.) are fine as long as the adapter matches wasmtime (see above).

**Provider tests can't execute on wasm32-wasip1.** Always use `--no-run` for test compilation checks. Tests that need to actually run should use `#[cfg(test)]` with host-target-compatible code only.

**The `[package.metadata.component]` sections in provider Cargo.toml files** are vestigial from `cargo component build`. They are unused by the current build pipeline but kept for documentation of the WIT world mapping.

**Container logs vs runtime logs.** `docker compose logs omnifs` shows stdout/stderr from the entrypoint. Runtime FUSE traces go to `/tmp/omnifs.log` inside the container. Check both when debugging.
